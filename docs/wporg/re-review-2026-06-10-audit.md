## [BLOCKER] undocumented-external-services — Guideline 6/7: external/third-party services not disclosed in a readme "== External services ==" section
- disposition: **fix-code+readme** | effort: M | occurrences: 26

### Web research
WordPress.org Detailed Plugin Guidelines, Guideline 6 ("Software as a Service must provide functionality of substance and be clearly documented in the readme... preferably with a link to the service's Terms of Use") and Guideline 7 (no tracking without consent; document how user data is collected/used in the readme, preferably with a privacy policy). The Plugin Review team enforces this through Plugin Check's "trademarks/phone-home" and a dedicated readme requirement: any plugin that makes outbound calls to a host the plugin does not control MUST carry a "== External services ==" section. The de-facto required, reviewer-accepted format (used by Jetpack, WP Mail SMTP, WooCommerce, Akismet, MonsterInsights, etc.) is, per service: (a) Service name + canonical URL; (b) what the plugin uses it for; (c) exactly what data is sent and on which user action / trigger; (d) a link to the service's Terms of Service AND Privacy Policy. Reviewers reject when ANY contacted external host is missing — they grep the source for wp_remote_*/curl/file_get_contents(URL) and cross-check against the readme, so the disclosure must be exhaustive, not just the headline service. Loopback / same-origin / WordPress.org-core-update traffic does not need disclosure. Sources: https://developer.wordpress.org/plugins/wordpress-org/detailed-plugin-guidelines/ (Guidelines 6 & 7); https://developer.wordpress.org/plugins/wordpress-org/how-your-readme-txt-works/ ; example real-plugin format e.g. WP Mail SMTP / Jetpack readme "== External services ==" blocks on wordpress.org.

## [HIGH] unsanitized-superglobals — Unsanitized superglobals ($_SERVER/$_COOKIE/$_GET) read without wp_unslash + sanitize_*
- disposition: **must-fix-code** | effort: M | occurrences: 24

### Web research
WordPress requires every read of an input superglobal ($_SERVER, $_COOKIE, $_GET, $_POST, $_REQUEST) to be unslashed THEN sanitized: the canonical pattern is sanitize_text_field( wp_unslash( $_SERVER['X'] ) ) for plain text, esc_url_raw( wp_unslash(...) ) for URLs, sanitize_key()/absint() for keys/ints. WP magic-quotes-slashes all of these superglobals (including $_SERVER), so wp_unslash() must come first, then a context-appropriate sanitizer. The review team treats $_SERVER and $_COOKIE as fully attacker-controllable: HTTP_HOST, HTTP_X_FORWARDED_FOR, HTTP_VIA, HTTP_USER_AGENT, REQUEST_URI and all cookies are client-supplied headers; SERVER_NAME/SERVER_SOFTWARE/SERVER_PROTOCOL are server-config but the WPCS WordPress.Security.ValidatedSanitizedInput sniff (which Plugin Check runs) flags ALL of them with no allowlist, emitting both InputNotSanitized and MissingUnslash. This is the exact sniff the reviewer's tool fired on. WPCS does NOT flag bare isset()/empty()/array_keys() of a superglobal (no value is consumed), which is why several $_COOKIE/$_GET touches in the drop-in are not in the count. Sources: developer.wordpress.org Plugin Handbook 'Common issues' and 'Securing input' (sanitize early, wp_unslash + sanitize_*), the WordPress-Coding-Standards ValidatedSanitizedInput sniff docs (deepwiki.com/WordPress/WordPress-Coding-Standards/3.1), and WPVIP 'Validating, sanitizing, and escaping'. For advanced-cache.php specifically: it is a drop-in loaded from wp-content/advanced-cache.php by wp-settings.php BEFORE plugins/pluggables, so wp_unslash()/sanitize_text_field() are NOT guaranteed loaded — established drop-in caches (WP Super Cache wp-cache.php, Hyper Cache advanced-cache.php) sanitize with raw-PHP (custom URI sanitizer + preg_replace allowlists) rather than WP helpers, and reviewers accept raw-PHP sanitization in that pre-WP context. Plugin Check, however, still statically flags the raw read, so the standard remedy even in a drop-in is to wrap each read in wp_unslash() guarded by function_exists() (or a tiny local unslash helper) plus the existing regex filtering, and add a precise phpcs:ignore with a justification so the automated gate passes.

## [MEDIUM] echoing-script-style-instead-of-wp-enqueue — Echoing &lt;script&gt;/&lt;style&gt; instead of wp_enqueue / wp_add_inline_* (output-buffer optimizer injections + a genuinely-fixable login-page style)

CANONICAL RULE: developer.wordpress.org says front-end assets must be loaded with wp_enqueue_script/wp_enqueue_style on the wp_enqueue_scripts hook, and small inline additions must use wp_add_inline_script/wp_add_inline_style ATTACHED TO AN ENQUEUED HANDLE. This is the accepted pattern for this plugin class; these are approved on wp.org. The correct disposition for those occurrences is to KEEP them inline, add the WordPress.WP.EnqueuedResources phpcs:ignore with a clear justification comment (matching what RumInjector already does), and add one readme/FAQ paragraph explaining the OB/cache-write-time architecture.

## [MEDIUM] rest-permission-callback-info-unbound — REST /info route authorizes with verify() (signature+replay only), not verifyCommand() — token not bound to this site (aud) or this endpoint (cmd)

### Web research
register_rest_route has REQUIRED permission_callback since WP 5.5 (make.wordpress.org/core 2020-07-22 "REST API changes in WordPress 5.5"); a missing/misspelled callback silently makes a route public, which "has happened multiple times in the wild" with "catastrophic" results. The canonical pattern for signed/token endpoints (Auth0/OAuth2 WP integrations, application-passwords issue #92, determine_current_user token patterns) is to verify the token AND bind it to the specific audience/endpoint — i.e. include and check an `aud` (API/site identifier) and an action/endpoint claim, returning WP_Error otherwise. ## [LOW] file_io_http_api_wp_filesystem — file_get_contents / readfile / fopen / fwrite / file_put_contents — remote vs local file IO
- disposition: **justify-in-reply** | effort: S | occurrences: 101

### Web research
The wp.org "Detailed Plugin Guidelines" (https://developer.wordpress.org/plugins/wordpress-org/detailed-plugin-guidelines/) do NOT contain any rule mandating WP_Filesystem or the HTTP API — there is no guideline number for this. It is a Plugin Check / reviewer best-practice preference, not an approval-blocking guideline. (1) REMOTE fetches: the canonical correct approach is the WP HTTP API — wp_remote_get()/wp_safe_remote_get() with wp_remote_retrieve_body()/_response_code() — never file_get_contents() on a URL, because PHP url-fopen wrappers (allow_url_fopen) are frequently disabled, bypass WP proxy/SSL settings, and offer no timeout/error handling. wp.org reviewers DO actively flag file_get_contents() pointed at a remote URL. Reputable approved backup/cache plugins (UpdraftPlus, WP Super Cache, W3 Total Cache, WPvivid) all stream large archives with fopen/fread/fwrite and emit cache bodies with readfile() in their advanced-cache drop-ins — this is established, approved practice.
