# Optimize-tab bug fixes: contract (2026-06-11)

Two user-reported bugs, root-caused by a 2-finder workflow. Fix specs below are binding.

## Bug 1

### Root cause
The gate flag is produced exclusively by WooFragmentsProbe::detectFromScriptRegistry(), called from PerfReporter::reportStats() — and reportStats() only ever runs in wp-cron (60s heartbeat, preload worker) or agent REST-command contexts (cache_enable, perf_config_update). WooCommerce never registers the `wc-cart-fragments` handle in any of those contexts: WC core skips its frontend-scripts include when DOING_CRON is defined or the request is a REST request, and registration is hooked on `wp_enqueue_scripts`, which never fires outside a front-end template render. So `wp_script_is('wc-cart-fragments','registered')` is false on 100% of probe executions, the agent reports `woo_theme_fragments_supported:false` on every 60-second heartbeat, the CP re-stamps the column false, and the web toggle is permanently disabled for every site and every theme — detection structurally cannot succeed. The robust full-HTML probe WooFragmentsProbe::detect($pageHtml) (which checks 'enqueued' + the `.widget_shopping_cart_content` selector in the rendered page) is dead code in production: it is only invoked from PHPUnit tests, never from the front-end output-buffer path where the page HTML actually exists.

### Fix spec (binding)
AGENT (primary, apps/agent) — move the authoritative probe to where it can actually succeed: a real front-end render. (1) In the existing ob_start pipeline (CacheWriter::handle, apps/agent/includes/cache/class-cache-writer.php:117ff, or the Optimizer stage list near class-optimizer.php:180 where WooFragmentsRuntime already runs), call WooFragmentsProbe::detect($buffer) on cacheable front-end requests, throttled via a transient (e.g. once per 12h, plus immediately when the option is unset) and only when class_exists('WooCommerce'). Latch asymmetrically to avoid flap-purges: persist TRUE immediately on any positive detection (PerfReporter::persistWooSupported(true,$cache)); persist FALSE only after N (e.g. 3) consecutive negative front-end probes on distinct URLs, since WooCommerce ≥7.8 enqueues wc-cart-fragments only on the cart page or when the classic mini-cart widget renders, so a single product page can be a false negative. Reset the option (back to 'unknown') on switch_theme/activated_plugin/deactivated_plugin — the Scheduler already hooks these (class-scheduler.php:147-149). (2) Make reportStats() (class-perf-reporter.php:130-138) stop probing in cron: read the persisted option tri-state (get_option(OPTION_WOO_FRAGMENTS_SUPPORTED, null)) and OMIT the field from the stats body when never probed — the CP handler already treats absence as 'leave unchanged' (agent_handler.go:128). Optionally keep detectFromScriptRegistry() as an opportunistic upgrade only when the current request is genuinely front-end (!is_admin() && !wp_doing_cron() && !defined('REST_REQUEST')). API (apps/api) — make 'unknown' representable and stop dropping reports: migration mNN alters site_perf_config.woo_theme_fragments_supported to DROP NOT NULL (NULL = not yet probed; existing false rows can be reset to NULL since every current false is untrustworthy) and ideally adds woo_fragments_probed_at timestamptz stamped by UpdateWooThemeFragmentsSupported; change UpdateWooFragmentsSupported (repo.go:205) to upsert/seed a default config row (or at minimum check rows-affected and log) so reports for never-configured sites are not silently lost; surface the field as *bool/nullable through model.go:136, dto.go:117, the OpenAPI schema and regenerated TS client. Run `sqlc generate` — never hand-edit perf.sql.go. WEB (apps/web) — three-state UX in WooCommerceSection.tsx: supported (toggle enabled, current positive copy); confirmed-unsupported (disabled, copy: 'Your active theme replaces the standard WooCommerce mini-cart, so cart-aware caching can't be enabled safely' — honest terminal state, no false promise of auto-flip); unknown/pending (disabled, copy: 'Checking your theme for cart-fragments support — this happens automatically the next time your store's pages are visited', optionally with a 'Check now' affordance that reuses the existing perf-config push to trigger an agent-side re-probe-on-next-render). Update types.ts:134 to `boolean | null`. The unknown state fixes the current UX where 'not yet detected' and 'genuinely absent' are indistinguishable.

### Risks
- Cache-purge thrash: persistWooSupported() purges the ENTIRE page cache on any value flip (class-perf-reporter.php:267-276); a per-URL-flappy probe (WC >=7.8 enqueues wc-cart-fragments only on cart/mini-cart pages) without the asymmetric latch + N-consecutive-negatives rule would repeatedly nuke the cache of busy stores
- Correctness is safety-critical here: a false POSITIVE lets cart-holding visitors be served shared cached shells whose mini-cart never repaints (the exact cross-shopper-confusion #169's conservative default avoided) — keep the two-signal full detect() as the only source of TRUE, never loosen to registry-only
- Migration/typegen ripple: making the column nullable changes sqlc models, the ogen OpenAPI schema (oas_schemas_gen.go already models it OptBool), and the generated TS client; per the standing sqlc-regen discipline, regenerate rather than hand-edit (a hand-synced perf.sql.go previously caused the v0.32.1 prod-down 500)
- Backward compat: pre-fix agents (<= current) keep sending false on every heartbeat and would instantly re-stamp false over a NULL reset — the CP should either ignore false from agents below the fix version (version header check) or accept that the tri-state UX only becomes accurate after the agent fleet updates
- Genuinely-unsupported modern setups exist: block themes using the Mini-Cart block use the Store API, not wc-cart-fragments, so the toggle will (correctly) stay disabled there — the new copy must not promise eventual availability
- Seeding a config row from the agent write path (to fix the silent 0-row UPDATE) must preserve tenant scoping/RLS (InAgentTx) and must not fight the operator GET-path defaultConfig synthesis (service.go:202-203) by creating divergent defaults

## Bug 2

### Root cause
The "Enable CDN" SettingRow in CdnSection auto-saves on flip: its onChange immediately calls save({ cdn_enabled: v }), and OptimizeTab's save() PUTs the FULL current config ({ ...config.data, ...patch }) — so on a never-configured site the body carries cdn_url: "". The server's validateConfig correctly rejects cdn_enabled=true with an empty cdn_url (domain.Validation -> HTTP 422). The user can never enter a URL first because SettingRow renders its children (including the CDN URL TextField) ONLY when `checked` — the input is hidden while the toggle is off, the failing PUT's optimistic rollback flips the toggle back off and re-hides the input, and the error surfaces only as a generic toast ("Could not save setting."). It's a chicken-and-egg: the only way to reveal the URL field is the very action that 422s and reverts.

- apps/api/internal/perf/service.go:223-228 — UpdateConfig calls validateConfig on every PUT
- apps/api/internal/perf/handler.go:69 — route: g.PUT("/perf/config", ..., h.putConfig); handler.go:179-204 — putConfig binds the full-config DTO and returns httpx.Error for domain errors
- apps/api/internal/domain/errors.go:159-160 — KindValidation maps to http.StatusUnprocessableEntity (422)
- apps/web/src/features/perf/optimize/CdnSection.tsx:55-62 — the Enable CDN SettingRow: onChange={(v) => save({ cdn_enabled: v })} (toggle auto-saves on flip)

### Fix spec (binding)
Web-only fix; keep server validation exactly as-is (it is correct). File: apps/web/src/features/perf/optimize/CdnSection.tsx (primary), apps/web/src/features/perf/components/SettingRow.tsx (one additive optional prop).

1) Draft-enable instead of save-on-flip when no URL exists. In CdnSection add local state `const [enabling, setEnabling] = useState(false)`. The toggle's onChange: if turning ON and config.cdn_url is empty -> setEnabling(true), do NOT call save; if turning ON and a stored cdn_url already exists -> save({ cdn_enabled: true }) as today (passes validation); if turning OFF -> setEnabling(false) and, only when config.cdn_enabled was true, save({ cdn_enabled: false }) — never blank cdn_url (the full-config PUT already preserves it).
2) Reveal + focus the URL input during the draft. Pass `checked={config.cdn_enabled || enabling}` to the SettingRow so the children slot opens (SettingRow.tsx:77 gate). Add an optional `inputRef`/autoFocus on the CDN URL TextField focused when `enabling` flips true. Optionally show a one-line hint in the slot: "Enter your CDN URL to finish enabling." If mixing draft state into `checked` is deemed unclean, instead add an optional `open?: boolean` prop to SettingRow that ORs with `checked` for the children gate — additive, default-unchanged for the other ~30 rows.
3) Single atomic save once the URL is valid. While `enabling`, the URL field's commit (blur/Enter, Field.tsx:191) — or a small "Enable CDN" button mirroring the existing "Save credentials" button pattern (CdnSection.tsx:122-131) — calls save({ cdn_enabled: true, cdn_url: url.trim() }) as ONE patch, then setEnabling(false) on success. OptimizeTab.save already supports multi-key patches (Object.keys(patch) at OptimizeTab.tsx:48) and usePerfConfig's onSuccess merge handles multi-key patchKeys (usePerfConfig.ts:88-94). Client-side validation mirrors service.go:349: parseable URL, scheme http/https, non-empty host; until it passes, the save action is disabled and an inline field error shows — the empty-URL 422 becomes unreachable.
4) Inline 422 surfacing. Add local field-error state under the CDN URL TextField. Have CdnSection receive the mutation failure for its own saves: extend OptimizeTab.save to also accept an optional onError callback (or return update.mutateAsync's promise) so CdnSection can set the inline error from err.message (e.g. server's "cdn_url must be a valid http(s) URL" if client and server ever disagree). Keep the existing global toast (usePerfConfig.ts:108) for all other rows unchanged; for this field, inline is primary.
5) Guard the already-enabled edit path too: when cdn_enabled is true, block committing an empty/invalid URL from the blur-commit at CdnSection.tsx:69 (inline error instead of firing a PUT that will 422).
6) Adjacent one-liner while in the file: CdnSection.tsx:32 seeds the cdnUrl draft once at mount; adopt NumberField's render-time resync pattern (Field.tsx:98-102) so the draft follows optimistic/rollback cache changes.

No API or agent changes. No OpenAPI changes (same PUT body shape).

### Risks
- SettingRow is shared by ~30 Optimize rows — any change to its children-gating must be additive (optional `open` prop or OR-ing at the call site) so other rows' reveal behavior is untouched
- Switch shows ON during the draft (`enabling`) while the persisted config is still cdn_enabled=false; if the user navigates away mid-draft nothing was saved — acceptable, but add the inline hint so the half-done state is legible; ensure isSaving("cdn_enabled") spinners don't show during the draft (no mutation in flight)
- Multi-key patch { cdn_enabled, cdn_url } relies on usePerfConfig's patchKeys inference (usePerfConfig.ts:88-94) — verified it handles multi-key sets, but a concurrent save on another row during the CDN PUT exercises the onSuccess merge path; existing 0.33.8 merge logic covers it
- Exposing per-save errors from OptimizeTab.save (callback or mutateAsync) must not swallow the global toast for other callers — keep toast behavior in onError as today and only ADD the inline channel for CdnSection
- Client-side URL validation must stay no-stricter-and-no-looser than service.go:349 (http/https scheme + host) or users will hit server 422s the inline validator said were fine (or be blocked on URLs the server accepts)
- Toggling OFF still PUTs the full config including the stored cdn_url — do not zero the local draft on disable or a later blur-commit could overwrite the stored URL with empty and trip validation when re-enabling
