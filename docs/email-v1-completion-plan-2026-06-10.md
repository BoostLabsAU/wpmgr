# Per-Site Email v1 Completion — Build Plan (v0.36.0, pinned contracts, 2026-06-10)

Source: 4-designer + synthesis workflow (org-propagation, multi-connection+failover, attachment metadata, alerts+digest). Companion to docs/per-site-email-smtp-plan-2026-06-10.md.

## Scope ruling

AREA 1 — Org-wide default propagation: ACCEPTED as spec'd, CP-only. Mechanism is a durable River job (kind email_org_config_propagate) enqueued best-effort after a successful UpsertOrgConfig, fanning out the existing sync_email_config command (zero wire change) to inheriting sites via errgroup (limit 8, 15s/site, 15m job timeout). Architect decisions locked: NO opt-out gate (CP reads already present the org config as inheriting sites' effective config — agents not enforcing it is the bug the TODO at service.go:237 describes); degraded sites INCLUDED as best-effort targets (push path is heartbeat-independent); result surfaced via structured logs + one new SSE event email.config_propagated, PUT response body unchanged. The web SSE toast + save-copy change is IN scope (it is small and the behavior change deserves visible feedback). CUT: per-site retry queue (recovery = next save / manual /email/sync / SendTest pre-sync), delete-propagation (no per-site delete endpoint exists), and a count-preview endpoint.

AREA 2 — Multi-connection + failover: OPTION A, decisively. Full named connections as an ADDITIVE child table (site_email_connection, FK→site_email_config ON DELETE CASCADE, per-connection encrypted secret + optional sender identity); the existing config row remains the implicit primary under reserved key 'default'. Option B is REJECTED: m59 already shipped mappings/default_connection/fallback_connection as key-based routing columns — A completes them, B would orphan them while adding parallel throwaway fields; the agent ProviderRouter already has the per-FROM and fallback seams so A's agent delta is wiring, not architecture; and B delivers failover without the actual FluentSMTP parity value (per-sender provider routing) that is in the LOCKED v1 scope. Architect decisions: operator-chosen slug keys (pending Q4 confirm), per-site UI only in v1 (service/API symmetric for org rows, no org UI), replace-all registry semantics with buildAgentConfigReq as the single full-state choke point. DEFERRED: per-connection bounce-webhook route tokens — v1 ships the documented primary-provider-only bounce ingest limitation with UI copy on the connection editor (Q2).

AREA 3 — Attachment metadata in the email log: ACCEPTED as spec'd, fully additive. Canonical shape [{"name":"<basename>","size_bytes":<int64>}] pinned at every layer; capture cap 50/message agent-side, ingest cap 100 with name sanitization (filepath.Base + 255 clamp); CP stores jsonb NOT NULL DEFAULT '[]'; lists carry attachment_count (jsonb_array_length), detail carries the array (mirrors the body list/detail split). Architect decisions: list-view paperclip+count chip on BOTH per-site and fleet tables (cheap, useful); mime NOT captured (privacy minimalism, pinned shape); attachment names NOT in free-text search. Its lone "m62" and "schema v11" claims are merged into the shared migration and shared agent bump below.

AREA 4 — Failure alerts + scheduled digest: ACCEPTED — instance mailer via mailer.Enqueuer, exactly the backup-notification pattern; zero agent changes. The dogfood-send-via-tenant-provider option is REJECTED for v1 (open-relay primitive through customer infra; alerts about a broken email stack must not depend on that stack; needs an agent command). New org-level tables email_notify_settings + email_alert_state (durable per-site throttle via single-statement conditional-UPDATE claim — NOT the in-memory throttle, NOT extending uptime-owned alert_configs). Per-failure alert hooks ONLY agent-ingested status=failed, strictly best-effort after IngestLogBatch; hourly DigestWorker with claim-advance-before-send. Architect decisions: digest-only for webhook bounces in v1 (Q2), skip zero-activity digests silently, Notifications card lives on /email (not /settings/alerts), default throttle 60m (range 15m–24h). GET notify-settings NEVER 404s (0.35.1 lesson). Ships degraded-but-honest when instance SMTP is unconfigured: instance_mailer_configured flag + UI warning banner; the prod SMTP ops action rides the release checklist (Q3).

CROSS-CUTTING CUTS/DEFERRALS for v0.36.0: per-connection webhook tokens; webhook-triggered per-failure alerts; org-level connections UI; "send alerts via your own provider" (explicit v2); attachment mime + search; per-site propagation retries; per-site config DELETE endpoint (when it lands, it must call SyncConfigToAgent so the org fallback pushes).

## Backward compatibility

OLD AGENT (0.35.x) vs NEW CP: (1) new sync_email_config keys (connections/default_connection/fallback_connection) are silently dropped by the KNOWN_KEYS whitelist (class-sync-email-config-command.php:35-47) — old agents keep single-provider behavior bit-for-bit; (2) mappings values become key STRINGS — old resolve_connection's is_array($mapped) check fails safely → falls to the primary row, never mis-routes (do NOT ship mappings as objects); (3) org-propagation pushes the IDENTICAL wire body old agents already accept; pre-0.35 agents 404 the command → counted failed + warn-logged, harmless; (4) old-agent log pushes omit connection_key/attachments → tolerant ingest coerces ''/[] → attachment_count 0, primary chip semantics, UI renders nothing; (5) re-push of the org config is idempotent on the agent (option upsert), so River retries with fresh jtis are safe. NEW AGENT (0.36.0) vs OLD CP (rollout window): encoding/json drops the unknown connection_key/attachments fields on the old ingest struct → 200 OK; the monotonic cursor advances so that window's metadata is never backfilled — acceptable for a log, and eliminated by the mandatory CP-before-agent deploy order; a 0.36 agent that never receives the new sync keys simply has an empty registry and routes via the primary exactly as 0.35. WEB SKEW: old web bundle vs new API ignores extra JSON fields and unknown SSE types (use-email-events startsWith('email.') + unknown-case fallthrough — email.config_propagated is additive-safe); new web vs old API (self-host skew): attachment_count/attachments/connections undefined → optional chaining renders nothing; the notify-settings hooks must treat a 404 from a pre-0.36 API as feature-unavailable and hide the card (the 0.35.1 useEmailConfig lesson — the NEW API itself never 404s that GET). DB/SCHEMA: every m62 statement is idempotent (IF NOT EXISTS + guarded DO blocks), purely additive, auto-applies on api boot; pre-existing site_email_log rows read as attachments '[]'/count 0/connection_key '' permanently (fine for a log, documented); agent dbDelta v10→v11 adds two columns only — a downgraded 0.35 agent against a v11 local table is unaffected (extra columns defaulted/NULL); IngestEmailLogEntry's attachments=EXCLUDED.attachments overwrite is benign (same-agent re-push only; cursor prevents downgrade re-pushes). API CONTRACT: attachment_count joins the OpenAPI required list — safe under servers-may-add-fields, but the generated TS types change, so packages/openapi-client regen ships in the SAME PR; PUT /email/org-config and the agent ingest response contract are byte-identical to 0.35. River: both new job kinds only ever enqueued by 0.36 code, so no orphan-kind risk during rollout.
