# M5.7 ‚Äî Organisations + per-site sharing + tenant-namespaced storage

Status: APPROVED for implementation (2026-05-31). Pre-open-source milestone.
Decisions confirmed by the user; full research in the `multiorg-sharing-research`
workflow output.

## Confirmed decisions
1. **Tenant = Organisation.** Keep the DB table `tenants`, `tenant_id` columns,
   and the `app.tenant_id` GUC unchanged ‚Äî "organisation" is a DTO/UI rename +
   UX (switcher, create-org, members CRUD) only. Membership is already N:N
   (`memberships` UNIQUE(user_id, tenant_id)). NO isolation-model change.
2. **Storage: keep dedup, add a browseable index.** Do NOT re-prefix chunks.
   Chunks stay `chunks/<tenant_id>/<blake3>` (content-addressed, per-tenant
   dedup via `UNIQUE(tenant_id, blake3)`). Add ONE per-snapshot index object at
   `tenant/<tenant_id>/site/<site_id>/backup/<snapshot_id>/manifest.json` at
   SubmitManifest time for the browseable tenant‚Üísite‚Üíbackup layout + per-site
   lifecycle. **No migration, no agent change.**
3. **Per-site sharing = outside collaborators.** Invite ANY email; on accept a
   `users` row is created/linked but NO `memberships` row ‚Äî the site-scoped
   restrictive RLS is their ONLY access.
4. **Invites = tokenized email accept links** (hash-stored, 7-day expiry,
   single-use, email-bound) ‚Äî mirrors `pairing_codes`. Uses the existing go-mail
   SMTP (M5). Keep the admin-set-password member-create path available too.
5. **Per-share optional expiry** (`expires_at`) covers time-limited support
   access; expired grants vanish via an RLS expiry filter (zero cleanup).
6. **Keep the 4-role vocab** (`viewer<operator<admin<owner`); add a DB CHECK.

## Crux: the RESTRICTIVE site-scope RLS

Postgres OR-combines PERMISSIVE policies, so a naive permissive site-share policy
would **widen** access. The site allowlist MUST be a **RESTRICTIVE** policy
(AND-combined ‚Üí can only ever remove rows), gated by a dedicated `app.site_scope`
GUC so it is a **no-op for normal members and all service paths**.

Per table that carries `site_id` (predicate keys on `id` for `sites`, on
`site_id` for children):
```sql
CREATE POLICY <t>_site_scope ON <t>
  AS RESTRICTIVE FOR ALL
  USING (
    coalesce(current_setting('app.site_scope', true), '') <> 'on'
    OR site_id = ANY (string_to_array(
         nullif(current_setting('app.allowed_site_ids', true), ''), ',')::uuid[])
  )
  WITH CHECK ( <same> );
```
- Normal members never set `app.site_scope` ‚Üí policy is a tautology ‚Üí full org
  access unchanged.
- Service paths (`app.agent`, `app._gc`, `app._scheduler`) never set it ‚Üí inert.
- The allowlist is **passed in the GUC** (resolved once at auth time via
  `site_shares_self_read`), so policies do NOT subquery `site_shares` ‚Üí no
  recursion. (If ever moved into the policy, use a SECURITY DEFINER
  `app_can_access_site()` owned by `wpmgr_app` ‚Äî NOT superuser/bypass.)

### RLS coverage ‚Äî MUST add the restrictive policy to ALL of these (21 direct):
`sites` (key on `id`), `update_tasks`, `backup_snapshots`, `backup_schedules`,
`backup_schedule_runs`, `restore_runs`, `site_alert_state`, `site_uptime_probes`,
`autologin_tokens`, `autologin_policies`, `agent_activity_log`, `agent_diagnostics`,
`agent_php_errors`, `agent_login_events`, `agent_nonces`, `site_destinations`,
`site_error_config`, `site_login_brand`, `site_security_config`, `scan_runs`,
`scan_findings`, `scan_run_hashes`.
*(agent_nonces is service-only but include for completeness/defense.)*

### Indirect children (no direct site_id) ‚Äî gate by parent:
- `backup_manifest_entries` (‚Üí snapshot_id), `restore_run_events` (‚Üí restore_run_id):
  add a RESTRICTIVE policy whose USING joins to the parent's `site_id`
  (`snapshot_id IN (SELECT id FROM backup_snapshots WHERE ‚Ä¶)`), OR guarantee the
  app only ever reaches them through a parent already RLS-gated in the scoped tx.
  Prefer the join-based restrictive policy for defense in depth.
- `backup_chunks`: tenant-scoped, content-addressed, shared across sites ‚Äî has NO
  site_id and MUST stay shared for dedup. A scoped user reaches a chunk only via
  a manifest of a snapshot they can already see; the CP presigns per-chunk for an
  authorized snapshot only. Do NOT add a site policy to `backup_chunks`; gate at
  the presign boundary (snapshot lookup inside the scoped tx).

## Data model

### Migration `apps/api/migrations/20260531050000_m19_orgs_sharing.sql`
(next after m18; copy header/idempotency + RLS-policy style from m16/m17.)

1. `ALTER TABLE sites ADD CONSTRAINT sites_id_tenant_key UNIQUE (id, tenant_id);`
   (backs the composite FK; free since `id` is PK ‚Äî prevents tenant drift.)
2. `ALTER TABLE memberships ADD CONSTRAINT memberships_role_check
    CHECK (role IN ('owner','admin','operator','viewer'));`
3. **`site_shares`** (the only structural grant table):
   ```
   id uuid PK DEFAULT gen_random_uuid()
   tenant_id uuid NOT NULL
   site_id   uuid NOT NULL
   user_id   uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE
   role      text NOT NULL DEFAULT 'viewer' CHECK (role IN ('viewer','operator','admin'))
   granted_by uuid REFERENCES users(id) ON DELETE SET NULL
   expires_at timestamptz NULL          -- NULL = durable; non-null = time-limited support
   created_at timestamptz NOT NULL DEFAULT now()
   UNIQUE (site_id, user_id)
   FOREIGN KEY (site_id, tenant_id) REFERENCES sites(id, tenant_id) ON DELETE CASCADE
   ```
   Indexes: `(user_id)`, `(tenant_id)`. RLS: ENABLE+FORCE;
   `site_shares_tenant_isolation` (tenant_id = app.tenant_id) for org admins;
   `site_shares_self_read` (FOR SELECT USING user_id = app.user_id) for the
   grantee's cross-org discovery. NO site_scope restrictive policy here (a scoped
   user reads their own shares via self_read; never lists others').
4. **`invitations`** (org + site, one table):
   ```
   id uuid PK, tenant_id uuid NOT NULL, email citext NOT NULL,
   scope text NOT NULL CHECK (scope IN ('org','site')),
   site_id uuid NULL REFERENCES sites(id) ON DELETE CASCADE,  -- set iff scope='site'
   role text NOT NULL,
   token_hash text NOT NULL UNIQUE,           -- sha256(token); never store token
   invited_by uuid REFERENCES users(id) ON DELETE SET NULL,
   expires_at timestamptz NOT NULL,           -- now()+7d
   attempts int NOT NULL DEFAULT 0,           -- rate-limit accept (mirror pairing_codes)
   accepted_at timestamptz NULL, accepted_user_id uuid NULL REFERENCES users(id),
   created_at timestamptz NOT NULL DEFAULT now()
   ```
   (requires `citext` extension ‚Äî `CREATE EXTENSION IF NOT EXISTS citext`.)
   RLS: ENABLE+FORCE; `invitations_tenant_isolation` (tenant_id=app.tenant_id);
   plus a pre-auth lookup policy `invitations_token_lookup`
   (FOR SELECT USING current_setting('app.invite_lookup', true) = 'on') for the
   public accept endpoint (mirror `api_keys_prefix_lookup`).
5. The RESTRICTIVE `<t>_site_scope` policy on all 21 direct tables + the
   join-based policies on the 2 indirect children, per the Crux section.

## Go control plane

### Principal + scoped tx
- `domain.Principal` += `Scope string` (`"org"`|`"site"`) and `AllowedSiteIDs []uuid.UUID`.
- `internal/db/db.go`: `InScopedTenantTx(ctx, tenantID, userID uuid.UUID, allowedSiteIDs []uuid.UUID, fn)` ‚Äî sets via `set_config(name,val,true)` (NEVER plain SET):
  `app.tenant_id`, `app.user_id`, `app.allowed_site_ids` (comma-joined),
  `app.site_scope='on'`. Normal members keep `InTenantTx`/`InTenantTxAsUser`.
- **Central dispatch (critical):** make the scoped principal's DEFAULT tx wrapper
  the scoped one ‚Äî repos/services should pick `InScopedTenantTx` vs `InTenantTx`
  based on `Principal.Scope`, NOT per-handler, so a forgotten path can't silently
  run unscoped (= full-org leak). Add a helper on the principal/ctx, e.g.
  `RunTenantTx(ctx, p, fn)` that branches on scope.

### middleware/auth.go ‚Äî scope resolution
After resolving the user + active tenant: if `RoleInTenant(user, tenant)` returns
a membership role ‚Üí `Scope="org"` (unchanged). Else look up
`GetActiveSharesForUserTenant(user, tenant)`; if ‚â•1 ‚Üí `Scope="site"`,
`AllowedSiteIDs=[site_ids]`, `Role=` highest per-site role (clamp `admin`‚Üí no
member-manage perms). Else ‚Üí no access to this tenant (403). A site-scoped
principal still passes `RequireTenant` (it has an active org).

### RequireSiteAccess(:siteId) middleware
Applied AFTER `RequirePermission` on per-site routes: no-op for `Scope="org"`;
for `Scope="site"` returns 404 when `:siteId ‚àâ AllowedSiteIDs` (clean 404, not a
silent empty list). Belt-and-braces in front of RLS.

### Endpoints (org)
- `POST /api/v1/orgs` ‚Äî create org + creator‚Üíowner membership (createTenant exists).
- `POST /api/v1/orgs/{orgId}/activate` ‚Äî verify membership via RoleInTenant ‚Üí
  `SetActiveTenant` (already built, currently UNWIRED) writes session
  active_tenant_id. (Keep X-Tenant-ID as the per-request escape hatch.)
- `GET /api/v1/members` (exists) + `PATCH /api/v1/members/{userId}` (role change)
  + `DELETE /api/v1/members/{userId}` ‚Äî both `PermMemberManage`, actor-role
  ceiling (reuse Service.Invite's `actorRole.AtLeast`). Cannot demote/remove the
  last owner.

### Endpoints (sharing)
- `GET /api/v1/sites/{siteId}/shares` (PermMemberManage or site admin) ‚Äî collaborators.
- `POST /api/v1/sites/{siteId}/shares` `{email, role, expires_at?}` ‚Äî if email is a
  known user ‚Üí insert site_shares; else create invitation(scope=site) + email link.
- `DELETE /api/v1/sites/{siteId}/shares/{userId}` ‚Äî revoke (immediate).
- `GET /api/v1/shared-with-me` ‚Äî any user; backed by `site_shares_self_read`
  (returns the sites + org names shared to the caller).
- `POST /api/v1/invitations/accept` `{token, email}` ‚Äî PUBLIC (session-creating):
  under `app.invite_lookup`, validate token_hash + email + expiry + single-use +
  rate-limit attempts; create/link user; scope=org‚Üímembership, scope=site‚Üísite_shares;
  mark accepted; start a session.
- Audit events (existing hash-chained `audit_log`, no schema change):
  `share.invited/accepted/granted/revoked`, `member.invited/role_changed/removed`,
  `org.created`, plus `support.granted/revoked/expired` (metadata: actor+grantee+expiry).

### Invite delivery ‚Äî COPY-LINK FIRST (email deferred)
**V1 primary path = copy-able accept link, NO email required.** The invite-create
response ALWAYS returns the raw accept link `${PUBLIC_BASE_URL}/accept?token=‚Ä¶`
(the token is shown once, like an API key) so the inviter can copy + share it
directly. The UI shows a "Copy invite link" button on every invite/share.

Email is an OPTIONAL, separate **notification engine** (its own
`WPMGR_NOTIFY_SMTP_*` config ‚Äî distinct from the M5 alert SMTP), added later once
a provider is chosen. When `WPMGR_NOTIFY_SMTP_HOST` is set, also email the link;
when empty (default now), the copy-link is the whole flow (no-op email, no error).
Do NOT couple invitations to the alert SMTP. Token is hash-stored; the raw link is
returned only in the create response and never persisted in plaintext.

### Storage index object
At `service.go` SubmitManifest (after the snapshot is finalized), write
`tenant/<tenant>/site/<site>/backup/<snapshot>/manifest.json` (the manifest JSON
already assembled) via the existing blobstore Put. Best-effort (a failure here
must not fail the backup). Chunk keys UNCHANGED. Presign paths for a scoped
principal MUST run the snapshot/chunk lookup inside `InScopedTenantTx` so RLS
denies non-granted sites before any URL is minted.

## Security review (MANDATORY before merge) ‚Äî checklist
1. The site-scope policy is `AS RESTRICTIVE` on EVERY one of the 21 tables +
   the 3 indirect children (backup_manifest_entries, restore_run_events,
   scan_run_hashes); none missed (assert coverage in a test).
2. All new GUCs set via `set_config(...,true)` / SET LOCAL only (pgBouncer-safe);
   no plain SET; no leak across pooled connections.
3. Every per-site read path for a scoped principal routes through
   `InScopedTenantTx` (central dispatch, not per-handler) ‚Äî no unscoped path.
4. Cross-org: a site-scoped principal provably sees ONLY the shared site across
   sites/backups/restores/updates/diagnostics/errors/activity/uptime/security/
   scans/autologin; isolation tests prove denial of every other site + org.
5. `EnforceRLSRole` (wpmgr_app NOSUPERUSER NOBYPASSRLS, boot hard-fail) intact;
   River/GC/scheduler/backup workers keep running as wpmgr_app with their service
   GUCs (never app.site_scope).
6. Invitation tokens: hash-only storage, single-use (accepted_at), 7d expiry,
   attempts rate-limit, accept bound to invited email (no identity swap/enum).
7. composite FK `site_shares(site_id,tenant_id)->sites(id,tenant_id)` ships in the
   same migration as `sites UNIQUE(id,tenant_id)` (no tenant drift).
8. Presign: a scoped user only ever gets per-chunk GETs for an authorized snapshot
   (lookup inside the scoped tx); never a bucket/tenant-wide presign.
9. Last-owner protection: cannot remove/demote the last org owner.
